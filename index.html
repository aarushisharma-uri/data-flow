üóÇÔ∏è Understanding the Data Structure

STAGING_PROD ‚Üí ADVANCEMENT ‚ñº

Purpose: Contains copies of tables from AWA (Advancement Web Application)

TMS_[...] Tables (Code & Description Files)

  • Purpose: Description and code lookup tables
  • Source: Oracle database (marked as views but function as tables)
  • Structure: In AWA, these may exist as one massive table, but they're broken down into separate TMS tables to make management easier
  • Usage Tip: When creating PowerBI dashboards, most of the time we use short descriptions

Non-TMS Tables (Actual Data Tables)

  • Tables without "TMS" prefix are actual data tables
  • ‚ö†Ô∏è Ignore these postfixes: Tables with "_PK" and "_ST" postfixes
    • These are subsets for pieces like addresses or allocations
    • Mike has never used them in practice

üîë ENTITY Table (Most Important)

Purpose: Contains constituent information
  • Key Details: Includes fields like ID, start date (ST_START_DT), end date (ST_END_DT) for dynamic record management
  • Source: Snowflake datasets which are updated nightly
  • Historical Tracking: Multiple data entries exist for each constituent because AWA tracks the complete history of constituent record changes
‚ö†Ô∏è CRITICAL - Identifying Current Records:

To get current/active constituent records, filter by:

ST_END_DT = Friday, December 31, 9999

Why? This date indicates a record is currently active. Historical records will have actual past dates.

Aspect ENTITY URI_ENTITY_SUMMARY

What is the difference between Views and Tables? ‚ñº
Aspect Table View
Definition A database object that stores data in rows and columns A virtual table representing the result of a stored query
Data Storage Physical storage of data Does not store data itself
Performance Generally faster for direct queries Query runs each time view is accessed
Updates Data persists until explicitly changed Always reflects current underlying data
üìù Note about TMS tables: In our system, TMS tables are marked as views in the source (Oracle) but function as tables for our purposes. You can treat them like regular tables when building reports.
What is the difference between a Column and a Measure? ‚ñº

üìä Columns

  • Definition: A field in a table that contains data
  • Nature: Static component that exists in the dataset
  • Data Level: Contains data at the row level - each row has a value for each column
  • Storage: Values are stored in the dataset and are part of the table's structure
  • Use Case: When you need to store calculated results at the row level
‚úÖ Column Example:

Adding a new column to calculate full name by concatenating first and last names:

Full Name = [First Name] & " " & [Last Name]

üìà Measures

  • Definition: A dynamic calculation used to perform aggregations or calculations on data
  • Nature: Dynamic - determined by the context of the report (filters, dimensions selected)
  • Storage: Not stored in the dataset - computed on the fly when needed
  • Use Case: For calculations that need to be dynamic and responsive to report interactions
  • Common Uses: Sums, averages, counts, and other aggregations
‚úÖ Measure Example:

Calculating total sales dynamically based on filters applied in a report:

Total Sales = SUM(Sales[Amount])

Quick Comparison

Aspect Column Measure
Calculation Time When data is loaded/refreshed When report is rendered
Storage Stored in model (uses memory) Not stored (calculated on demand)
Context Aware No - same value regardless of filters Yes - changes based on filters/slicers
Can be used in Rows, filters, slicers, axes Values area of visualizations
Row-level data ‚úÖ Yes ‚ùå No (aggregated)
‚ö†Ô∏è When to use which:
  • Use a Column when you need to categorize, filter, or slice data, or when you need row-level calculations
  • Use a Measure when you need aggregations that respond to user interactions (filters, slicers)
Why do I see multiple records for the same constituent? ‚ñº

AWA (Advancement Web Application) tracks the complete history of constituent record changes. This means:

  • Every time a constituent's information changes (address, name, status, etc.), a new record is created
  • The old record gets an end date (ST_END_DT) set to when the change occurred
  • The new record gets a start date (ST_START_DT) of when it became active
  • Current/active records always have ST_END_DT = 12/31/9999
‚úÖ To get only current records:

Filter your data where:

ST_END_DT = "12/31/9999"
